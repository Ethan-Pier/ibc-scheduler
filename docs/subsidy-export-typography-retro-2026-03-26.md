# 补贴导出字体问题复盘

这次“补贴导出仍显示 `Calibri`”的问题，表面看是字体没改干净，实际暴露出的是对 Excel 模板、OOXML 字体记录和静态资源缓存三层机制判断不够彻底。

## 这次犯的错误

### 1. 把 `font.name = 宋体` 误当成问题已解决

一开始我用 `openpyxl` 读模板和导出文件，看到正文单元格的 `font.name` 已经是 `宋体`，就以为字体问题已经修好。

但用户在 WPS/Excel 工具栏里选中正文数字单元格时，仍然显示 `Calibri`。这说明：
- 表层 API 读到的字体名称，不等于 Office 实际采用的完整字体定义
- 不能只看 `font.name`

### 2. 没有第一时间检查底层 `styles.xml`

真正的根因不在单元格文本本身，而在 `.xlsx` 里的字体记录：
- 一些正文样式虽然名字已经是 `宋体`
- 但底层字体对象还残留 Office 主题字体属性，例如 `scheme=minor`

这会导致：
- 代码侧看起来像“宋体”
- Office/WPS 工具栏仍把它判成 `Calibri`

以后遇到类似问题，应该更早直接检查：
- `xl/styles.xml`
- `cellXfs` 对应的 `fontId`
- 字体对象里是否残留 `scheme/theme/color`

### 3. 先改了“可见样式”，没同步考虑“Office 的字体判定规则”

我前两轮修复做了这些事：
- 把模板可见区域字体字号对齐到参考工资单
- 把富文本改成普通文本
- 修正合并单元格区域的样式残留

这些都不是错，但还不够。

因为这只能保证“看起来像宋体”，不能保证 Office 工具栏对字体的识别结果。真正需要的是：
- 重建成“纯宋体”字体对象
- 不再沿用旧模板残留的主题字体属性

### 4. 忽略了静态模板文件缓存

补贴导出模板走的是固定 URL：
- `subsidy-template.xlsx`

这意味着只要文件名不变，浏览器就可能继续复用旧模板缓存。结果就是：
- 本地模板已经修好
- 线上用户却还可能导出旧版文件

最后必须补上的不是“再部署一次”，而是：
- 给模板 URL 加版本戳，强制拿新模板

### 5. 对 Excel 模板问题分层不够清楚

这次问题其实分成三层：
- 模板可见样式是否对齐参考表
- 导出逻辑是否会把模板样式正确保留下来
- 最终 OOXML 字体对象是否仍残留主题字体属性

我前面几次修复是逐层补上的，但一开始没有把这三层拆开，所以来回了几轮。

## 这次最终确认的根因

最终根因是两件事叠加：

1. 模板中部分字体对象残留 Office 主题字体属性，导致 WPS/Excel 工具栏仍显示 `Calibri`
2. 线上前端继续命中旧的 `subsidy-template.xlsx` 缓存，导致用户导出的其实不是最新模板

## 这次最终采用的正确做法

- 参考 2025-11 工资单逐区块校准补贴模板的字体、字号、字重、对齐
- 将“发放事由”从手工富文本改为普通文本，避免再引入额外字体定义
- 重建模板字体对象，只保留真正需要的宋体属性，去掉主题字体残留
- 用本地脚本模拟真实导出，检查最终导出件里所有可见单元格是否仍存在非宋体
- 给模板 URL 加版本戳，解决线上缓存

## 下次处理 Excel 模板的正确顺序

1. 先确定问题发生在：
   - 模板
   - 导出逻辑
   - 还是最终生成的 OOXML
2. 不只看 `openpyxl` / `exceljs` 读出来的 `font.name`
3. 直接检查：
   - `styles.xml`
   - `fontId`
   - `scheme/theme`
4. 生成一份临时导出文件，检查真实输出，而不是只看模板
5. 若模板走固定静态 URL，发布时同时处理缓存问题

## 以后遇到同类问题的最小检查表

- 用户看到的问题，是视觉问题还是 Office 工具栏识别问题
- 模板字体对象里是否残留 `scheme=minor` 或主题色
- 导出后真实文件里是否还有非目标字体
- 线上静态模板是否被旧缓存命中
- 是否需要在部署时同步改模板 URL 版本
